## Overview

The very first commits. Fluxy's core identity was established here: an AI agent (Claude) running locally, accessed through a chat bubble, with an onboarding wizard that sets up the portal password and claims a unique handle.

---

## Foundation

### Claude SDK Integration

The Claude Agent SDK was wired in as the AI backbone (`claudesdk ready`). This established the pattern that defines Fluxy:
- Claude is the agent, not a stateless chatbot
- The agent has persistent memory (files), tools (bash, read, write), and a workspace
- Communication happens through the chat bubble — the agent's only channel to the human

### Onboarding Wizard

A multi-step wizard that runs on first access and sets up:
1. **Portal password** — protects access to the chat interface
2. **Handle** — the URL slug for the instance (`fluxy.bot/yourname`)
3. **Wizard prefill** — pulls suggested values from existing system config where possible

The wizard was designed so that every new Fluxy starts from a clean, intentional setup rather than bare defaults.

### Handle System

Each Fluxy instance gets a **handle** — a unique identifier used in:
- The public URL (`fluxy.bot/<handle>`)
- The relay connection for tunneled access
- Instance identification in multi-tenant contexts

Handle ownership is verified during onboarding (the handle must be available and pass format validation).

### Portal Password Change

The ability to change the portal password was implemented from day one — accessible from the settings panel once logged in.
